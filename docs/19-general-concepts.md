# General Concepts & Conventions

The foundational conventions every XenForo 2 add-on relies on: how code is
autoloaded, how classes are named and namespaced, how short class names resolve
**by context**, how you extend core classes, and how to type-hint factory-created
objects so your IDE (and your future self) understands them.

> **Sources (official XenForo docs):**
> - https://docs.xenforo.com/devs/general-concepts
> - https://docs.xenforo.com/devs/

> **XF2 only.** Everything here is XenForo 2.x (targeting 2.3+). XF1 patterns —
> `library/`, `DataWriter`, `_Model` classes, `Zend_*`, and the `load_class`
> events — do **not** exist in XF2 and must never be used. If you are porting an
> XF1 add-on, treat this page as a reset of your mental model.

---

## Where this fits

This doc is the "rules of the road." It assumes you can already create an add-on
skeleton (see `docs/02-addon-structure.md`) and build/export it (see
`docs/14-build-release-devtools.md`); it does **not** repeat those. The pre-requisite
knowledge XenForo assumes of you is PHP and MySQL (or similar) — prior experience
with XenForo or another framework is an advantage but not essential.

## Vendor components

XF2 is **not** built on a single overarching framework the way XF1 was. Instead it
pulls in a handful of popular, well-tested, open-source packages for specific jobs:

| Package | Used for |
|---|---|
| [SwiftMailer](https://github.com/swiftmailer/swiftmailer) | Email sending |
| [Guzzle](https://github.com/guzzle/guzzle) | HTTP client |

All third-party projects live in the `src/vendor` directory.

> **Gotcha for add-on developers:** it is **not currently possible** to add your
> own dependencies into `src/vendor`. You cannot drop in an arbitrary Composer
> package and expect XF's autoloader to pick it up there. Plan around the libraries
> XenForo already ships, or bundle and load your own code from within your add-on's
> own directory under `src/addons`.

## The autoloader

XenForo uses an autoloader that is **automatically generated by Composer**. This
means all XF code, all third-party vendor code, and all add-on developer code is
included throughout the project automatically — you never write `include`/`require`
for your classes.

The autoload root for **all** add-ons is the `src/addons` directory. Every class
name is therefore relative to that base location.

### Strict "class per file" naming

XenForo enforces a strict **one class per file** convention, and the class name
must identify the **exact** location of the file on disk:

- Each `.php` file contains exactly one class.
- The fully-qualified class name maps directly to the path under `src/addons`.

| Class name | File on disk |
|---|---|
| `Demo\Setup` | `src/addons/Demo/Setup.php` |
| `Demo\Entity\Thing` | `src/addons/Demo/Entity/Thing.php` |

Here `Demo` is the **add-on ID** and the first path segment. Read a class name and
you know the file; read a file path and you know the class name. There is no
configuration to maintain — the mapping is the convention.

> **For add-on developers:** this is why your add-on ID becomes the top-level
> namespace of every class you write. Pick it carefully at creation time (see
> `docs/02-addon-structure.md`); renaming it later means moving every file and
> rewriting every namespace.

## Namespaces

XF uses [PHP namespaces](https://www.php.net/manual/en/language.namespaces.rationale.php)
throughout so classes in the same namespace can reference each other succinctly.
It is recommended that all add-ons use namespaces too.

A class conceptually called `Demo\Entity\Thing` is, in the file itself, declared as
class `Thing` with the namespace `Demo\Entity`:

```php title="src/addons/Demo/Entity/Thing.php"
<?php

namespace Demo\Entity;

class Thing
{

}
```

Because the namespace is `Demo\Entity`, another class in that same directory — say
`AnotherThing` (`src/addons/Demo/Entity/AnotherThing.php`) — can be referenced from
inside `Thing` simply as `AnotherThing`, with no leading namespace, because they
share the `Demo\Entity` namespace.

> Classes from a **different** namespace must be referenced by their fully-qualified
> name (e.g. `\XF\Entity\User`) or pulled in with a `use` statement at the top of
> the file.

## Short class names (context-sensitive resolution)

This is one of the most important — and most frequently mishandled — XF conventions.
Many classes in XF are referenced by a **short class name** rather than their full
class name. A short name looks like `XF:User` or `Demo:Thing`. The full class it
resolves to is **entirely context-sensitive** — the *same* short name resolves to
*different* full classes depending on which factory method you pass it to.

Take `XF:User`. In the context of an **entity** call it resolves to
`XF\Entity\User`:

- The `XF` part is the file path root (the add-on ID — here, XenForo core).
- The `Entity` part is **implied by the type of call** (you're asking for an entity).
- The `User` part is the specific class.

The same principle applies to your own add-on's classes. Consider a `Thing` short
name in the `Demo` add-on:

```php
// Entity context -> Demo\Entity\Thing
\XF::em()->create('Demo:Thing');

// Repository context -> Demo\Repository\Thing
\XF::repository('Demo:Thing');
```

Notice the short class name `Demo:Thing` is **identical** in both calls. Only the
**call context** decides whether you get `Demo\Entity\Thing` or
`Demo\Repository\Thing`.

| Short name | Call context | Resolves to |
|---|---|---|
| `XF:User` | `\XF::em()->find(...)` / `create(...)` (entity) | `XF\Entity\User` |
| `Demo:Thing` | `\XF::em()->create('Demo:Thing')` (entity) | `Demo\Entity\Thing` |
| `Demo:Thing` | `\XF::repository('Demo:Thing')` (repository) | `Demo\Repository\Thing` |

> **Mental model:** a short name is `<addonId>:<ShortName>`. The middle segment of
> the *full* class (`Entity`, `Repository`, …) is supplied by the factory you call,
> never written into the short name itself. This is why the same string can mean two
> different classes — and why you should never try to "spell out" the middle segment
> inside a short name.

The entity/finder/repository factory methods are covered in
`docs/03-entities-finders-repositories.md`.

## Extending classes

A great many classes in XenForo are **extendable**, letting you override core
behaviour without editing core files directly.

### The XF1 way (removed)

In XF1 you would: create a Listener PHP file, write a class extending the original,
write a callback matching a `load_class` event signature that injected your class
name, then register a "Code event listener" in the Admin CP. **XF2 has removed the
`load_class` events entirely** in favour of a dedicated system called
**Class extensions**.

### The XF2 way — Class extensions

The process is reduced to two steps:

1. Create a class that will ultimately extend the original class.
2. Add a **Class extension** in the Admin CP specifying the base class and your
   extending class.

This cuts the boilerplate and gives a dedicated UI for viewing and managing
extensions.

### Worked example — extending the public Member controller

After creating a `Demo` add-on (via `xf-addon:create`, see
`docs/14-build-release-devtools.md`), place the extended class where it clearly
identifies (a) the owning add-on, (b) the type of class being extended, and (c) the
class name. For the public `Member` controller that path is
`src/addons/Demo/XF/Pub/Controller/Member.php`:

```php title="src/addons/Demo/XF/Pub/Controller/Member.php"
<?php

namespace Demo\XF\Pub\Controller;

class Member extends XFCP_Member
{

}
```

> **You extend `XFCP_Member`, not `XF\Pub\Controller\Member` directly.** This is the
> **XenForo Class Proxy** system (XFCP). It builds an **inheritance chain** so a
> single class can be extended by **multiple** add-ons at once without them
> clobbering each other. The convention is always: reference a dummy class that is
> the **current class name** (`Member`) prefixed with `XFCP_`.

Register the extension in **Admin CP > Development > Class extensions > Add class
extension**:

| Field | Value |
|---|---|
| Base class name | `XF\Pub\Controller\Member` |
| Extension class name | `Demo\XF\Pub\Controller\Member` |

The extension is now active but does nothing yet. To make it act, either **override**
an existing method (declare a method with the same name) or **add a new** one. Adding
a new action:

```php title="src/addons/Demo/XF/Pub/Controller/Member.php"
<?php

namespace Demo\XF\Pub\Controller;

class Member extends XFCP_Member
{
    public function actionHelloWorld()
    {
        return $this->message('Hello world!');
    }
}
```

Visit `index.php?members/hello-world` (relative to your board URL) and you'll see
the "Hello world!" message.

> Overriding an **existing** method (e.g. naming yours `actionIndex()` instead) does
> replace the original — but doing it bluntly is **not** the correct way to modify an
> existing action reply. The proper override technique (calling the parent through
> the proxy chain) and full coverage of controllers, actions and replies live in
> `docs/04-controllers-routing.md`. The events, listeners and class-extension system
> are covered in depth in `docs/12-events-listeners-extensions.md`.

## Type hinting

Many XF objects are created through **factory methods**. Convenient and consistent —
but your IDE usually can't infer the concrete return type.

```php
$repo = \XF::repository('Demo:Thing');
```

As far as the IDE knows, `\XF::repository()` returns an `XF\Mvc\Entity\Repository`.
That's partly useful, but the specific `Demo\Repository\Thing` object likely has many
methods the IDE can't see — so you lose autocomplete and argument hints on `$repo`.

Fix it with a one-line `@var` docblock immediately above the call. The syntax is
supported by most IDEs and many "PHP aware" editors:

```php
/** @var \Demo\Repository\Thing $repo */
$repo = \XF::repository('Demo:Thing');
```

The IDE now treats `$repo` as a `Demo\Repository\Thing` and gives full method/argument
completion.

> **For add-on developers:** make these docblocks a habit on every factory call —
> `\XF::em()->find()`, `\XF::repository()`, `\XF::service()`, `\XF::app()->finder()`,
> and similar. They cost one line and pay for themselves in correctness, and they are
> the single cheapest defence against calling a method that doesn't exist on the type
> you assumed.

### Type hinting inside class extensions (`extension_hint.php`)

Type hinting matters **especially** when extending classes. Your extended class does
not really extend the original — it extends a proxy that **does not actually exist**
as a concrete class (e.g. `XFCP_Member`). So your IDE has nothing to resolve `$this`
against, and can't autocomplete methods inherited from the real `Member` controller
or its parents.

To fix this, XenForo **automatically generates** a file named `extension_hint.php` and
stores it in your add-on's `_output` directory. It adds a reference the **IDE can read
but PHP can't**, so that inside any method of your extended class the IDE understands
what `$this` is and can suggest/autocomplete methods and properties from the `Member`
controller (or one of its parents).

| Aspect | Detail |
|---|---|
| File | `extension_hint.php` |
| Location | the add-on's `_output` directory |
| Generated | automatically by XenForo |
| Read by | the **IDE** (PHP itself ignores it) |
| Purpose | restore autocomplete for `$this` in `XFCP_`-extended classes |

> **Permissions note:** because `_output` is written to by tooling, the CLI user and
> your web-server user both need to be able to write there. If the file isn't
> appearing or updating, check write permissions on the add-on's `_output` directory
> (running your web server as your CLI user causes the least friction in
> development). General build/export workflow is in `docs/14-build-release-devtools.md`.

## Quick reference

| Concept | Rule |
|---|---|
| Autoload root | `src/addons` |
| File ↔ class | strict one-class-per-file; path mirrors the FQCN |
| Namespace | top segment = add-on ID; same-namespace classes need no prefix |
| Short name | `<addonId>:<ShortName>`; middle segment supplied **by call context** |
| `Demo:Thing` (entity) | `Demo\Entity\Thing` |
| `Demo:Thing` (repository) | `Demo\Repository\Thing` |
| Extend a class | create `XFCP_`-extending class, register a **Class extension** in Admin CP |
| `load_class` events | **removed in XF2** — do not use |
| Own `src/vendor` deps | **not** currently possible |
| Type hint a factory result | `/** @var \Full\Class\Name $x */` above the call |
| Extension autocomplete | auto-generated `_output/extension_hint.php` |

## Checklist

- [ ] Class file path exactly mirrors its fully-qualified name under `src/addons`.
- [ ] One class per file; namespace top segment matches the add-on ID.
- [ ] Short class names written as `<addonId>:<ShortName>` — no `Entity`/`Repository`
      segment baked into the string.
- [ ] Extended classes extend `XFCP_<ClassName>`, never the base class directly.
- [ ] Every class extension is registered in Admin CP > Development > Class extensions.
- [ ] No reliance on `load_class` events or any XF1 pattern (`library/`, `DataWriter`,
      `_Model`, `Zend_*`).
- [ ] No attempt to add custom dependencies to `src/vendor`.
- [ ] Factory calls carry a `/** @var ... */` docblock for IDE accuracy.
- [ ] `_output` is writable so `extension_hint.php` can be (re)generated.
